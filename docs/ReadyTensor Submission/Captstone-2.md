# Module 2 Capstone — Deploy and Monitor Your Fine-Tuned LLM

> Captured: 2026-05-06. Refer to the live URL for the canonical version.

## 1. Project Objectives

Create a **detailed deployment plan for an LLM of your choice**. Demonstrate
the ability to:

- Define a clear use case and requirements for an LLM application
- Select and justify an appropriate model and configuration
- Design a deployment strategy with infrastructure choices
- Plan for monitoring, observability, and cost management
- Communicate technical decisions clearly and professionally

## 2. Essential Requirements

### Publication Submission

The publication must include **six core sections**:

1. **Use Case Definition** — Problem statement, target users, input/output
   examples, success criteria, traffic expectations.
2. **Model Selection & Configuration** — Model name, source, parameter count,
   quantization approach, context length, max output tokens, with
   justification for each choice.
3. **Deployment Strategy** — Platform selection with rationale, hardware
   specifications, scaling approach, endpoint type, geographic region.
4. **Cost Analysis** — Monthly cost breakdown (compute, storage, network,
   monitoring), optimization strategies (caching, quantization, batching,
   spot instances, auto-scaling), cost-per-request calculation.
5. **Monitoring & Observability Plan** — Metrics to track (latency, error
   rate, throughput, token usage, GPU utilization), tools selection, alerting
   strategy.
6. **Security Considerations** — API authentication, rate limiting, input
   validation, PII handling, access control.

### GitHub Repository Submission

- **Deployment scripts** (at least one): Modal, vLLM, SageMaker, Docker
  Compose, Hugging Face, or equivalent.
- **Client code**: request script / notebook with example prompts and error
  handling.
- **Documentation**: README with deployment steps, environment setup, usage
  instructions.

## 3. Optional / Bonus Enhancements

- Architecture diagrams showing deployment design.
- Cost calculation spreadsheet with detailed estimates.
- Sample monitoring dashboard mockups.

## 4. Submission Deliverables

1. Ready Tensor publication meeting **≥ 70 %** of the
   *Technical Evaluation Rubric for technical publications*.
2. GitHub repository meeting **≥ 70 % of the "Essential" level** in the
   repository evaluation rubric.
3. Supporting materials (optional).

## 5. Submission Deadlines

Monthly review cycles, 11:59 PM UTC:

- January 05, 2026
- February 02, 2026
- March 02, 2026
- April 06, 2026
- May 04, 2026

If a deadline is missed, the project rolls over to the next month's review.
Reviews typically take ~2 weeks.

## 6. Pass Thresholds

- Publication: **≥ 70 %** of the Technical Evaluation Rubric.
- Repository: **≥ 70 %** of the "Essential" level in the repo rubric.
